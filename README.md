# NovaVec

> A production-quality vector database built entirely from first principles in C++20, with a Python application layer.

## One-line summary

NovaVec is a vector database that implements HNSW graph construction, IVF clustering, AVX2 SIMD distance kernels, CRC32-protected WAL durability, mmap-based immutable segment storage, and a selectivity-based query planner entirely from scratch in C++20, exposed to Python via pybind11 with zero-copy numpy interop and a Qdrant-compatible FastAPI REST layer.

---

## What makes this different from existing projects

- **Selectivity-based query planner**: Instead of always pre-filtering or always post-filtering, a `QueryPlanner` estimates filter selectivity from a CRoaring inverted index and chooses the cheaper strategy — bitmap-masked graph traversal (pre-filter) for rare values, overfetch-then-discard (post-filter) for common values. Threshold defaults to 20%.
- **IVF-HNSW hybrid index**: Each IVF Voronoi cell contains its own HNSW graph. At search time, `nprobe` cluster searches run in parallel via `std::async`, giving sub-linear traversal within each cell instead of IVF's flat linear scan.
- **WAL durability with three fsync levels**: A binary append-only log with CRC32 per-entry checksums provides crash recovery. `SYNC_ALWAYS` fsyncs every write; `SYNC_PERIODIC` uses a background thread (200 ms interval); `SYNC_NEVER` maximizes throughput at the cost of durability.
- **Qdrant-compatible REST API**: FastAPI endpoints follow Qdrant's request/response schema shape — `PointStruct`, `FilterCondition`, `must` conditions — so migration from Qdrant requires minimal client changes.
- **Recall-latency Pareto benchmarks**: `bench_runner.cpp` loads ann-benchmarks HDF5 datasets, builds the index, runs queries, and produces p50/p95/p99 latencies and Recall@10 — the standard metric for ANN evaluation.
- **RAG demo**: A FastAPI app backed by NovaVec retrieves Wikipedia passages and generates grounded answers via GPT-4o-mini or local Ollama. One command runs it end-to-end.
- **AVX2 SIMD distance kernels**: L2 and dot-product kernels process 8 floats per cycle using 256-bit YMM registers with fused multiply-add (`_mm256_fmadd_ps`). For cosine, vectors are normalized at insertion so cosine distance reduces to `1 - dot(a,b)`, avoiding sqrt and division at query time.

## Key design decisions

- **Flat contiguous `float*` array for vector storage** vs. `std::vector<float>` per node. *Alternative rejected*: per-node vectors scatter data across the heap, destroying cache locality during distance computation. *Reasoning*: A flat array means vector i is always at `data + i * dim` — a single pointer arithmetic operation, and all vectors in a segment are in one contiguous allocation that benefits from CPU prefetch.

- **Generation counter visited set** vs. `std::unordered_set<int>` per query. *Alternative rejected*: allocating and clearing a hash set for every query costs O(ef) heap allocations plus a clear() call. *Reasoning*: A shared `std::vector<uint32_t> visited_` and a per-query `current_gen_` counter make the visited check O(1) with zero allocation — comparing `visited_[i] == gen` instead of `set.count(i)`.

- **Diverse neighbor heuristic (Algorithm 4)** vs. naive k-nearest. *Alternative rejected*: selecting the k closest candidates clusters neighbors in one direction, leaving the node weakly connected to the broader graph — recall degrades at high k. *Reasoning*: The heuristic enforces that each new neighbor adds a new "direction" from the query, ensuring the local neighborhood spans multiple graph regions and preserves long-range recall.

- **CRoaring bitmaps for metadata inverted index** vs. `std::vector<DocId>` per field value. *Alternative rejected*: linear vector scan for bitmap AND is O(|A| + |B|) but non-compressed, wasting memory for dense fields. *Reasoning*: CRoaring uses run-length encoding for dense ranges and sorted arrays for sparse sets, with SIMD-accelerated AND operations — the same data structure Lucene uses for its filter cache.

- **Selectivity-based query planner** vs. always pre-filter or always post-filter. *Alternative rejected*: always pre-filtering is expensive for high-selectivity filters (matching 80% of vectors) because it limits graph exploration unnecessarily; always post-filtering is expensive for low-selectivity filters because it requires fetching 20× more candidates. *Reasoning*: The 20% threshold balances these extremes — below it, pre-filter wins; above it, post-filter overfetch is bounded at ≤5×.

- **WAL with configurable fsync levels** vs. always-sync or never-sync. *Alternative rejected*: always-sync limits write throughput to ~500 ops/s on a 2 ms disk; never-sync loses data on power failure. *Reasoning*: SYNC_PERIODIC (200 ms background thread) gives ~200 ms maximum data loss on crash while allowing batched writes at disk bandwidth.

- **mmap for immutable segments** vs. `read()` into a buffer. *Alternative rejected*: read() into a user-space buffer doubles memory usage (kernel page cache + user buffer) and requires explicit management. *Reasoning*: mmap maps the file directly into virtual address space — the OS page cache is the buffer, `madvise(MADV_RANDOM)` disables useless readahead, and large sequential scans benefit from huge-page backing without code changes.

```bash

git clone https://github.com/yourname/novavec && cd novavec

# 2. Install Python dependencies
```

## Python API

```python
import novavec
import numpy as np

# Create a collection (384-dim cosine similarity)
col = novavec.Collection(
    name="demo",
    dim=384,
    metric="cosine",
    index_type="hnsw",
    M=16,
    ef_construction=200,
)

# Upsert vectors with metadata
col.upsert(1, np.random.rand(384).astype(np.float32), {"category": "science"})
col.upsert(2, np.random.rand(384).astype(np.float32), {"category": "history"})
col.upsert(3, np.random.rand(384).astype(np.float32), {"category": "science"})

# Search — no filter
results = col.search(np.random.rand(384).astype(np.float32), top_k=5)

# Search — with metadata filter (pre/post-filter chosen automatically)
results = col.search(
    np.random.rand(384).astype(np.float32),
    top_k=5,
    filter={"category": "science"},
)

for r in results:
    print(r.id, r.score, r.metadata())

# Persist to disk
col.save()

# Reload
col2 = novavec.load_collection("./demo_novavec_data")
print(col2)  # novavec.Collection(name='demo_novavec_data', size=3, dim=384)
```

## REST API

```bash
# Create collection
curl -X POST http://localhost:8000/collections/my-docs \
     -H "Content-Type: application/json" \
     -d '{"dim": 384, "metric": "cosine", "index_type": "hnsw"}'

# Upsert vectors
curl -X POST http://localhost:8000/collections/my-docs/points \
     -H "Content-Type: application/json" \
     -d '{"points": [{"id": 1, "vector": [...], "payload": {"tag": "ml"}}]}'

# Search
curl -X POST http://localhost:8000/collections/my-docs/points/search \
     -H "Content-Type: application/json" \
     -d '{
       "vector": [...],
       "top_k": 5,
       "filter": {"must": [{"key": "tag", "match": {"value": "ml"}}]}
     }'
```
